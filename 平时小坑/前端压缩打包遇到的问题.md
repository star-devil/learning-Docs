## 无法请求视频
### 浏览器对视频文件的特殊处理

当浏览器加载<video>标签的视频文件时，它会：

- 要求 identity 编码（即不压缩）

- 使用 Range 请求来支持视频控制功能

原因：

- 🎯 支持范围请求：视频需要支持拖拽、快进、暂停时的精确定位

- 🗜️ 避免双重压缩：MP4本身已经是高度压缩的格式

- ⚡ 性能考虑：避免服务器端解压缩的开销



## 通过接口回去到的dicom文件无法展示

###  问题分析
错误信息显示：
1. WASM 文件的 MIME 类型不正确
2. WASM 文件被 gzip 压缩了，但浏览器无法正确处理
3. 魔术字节 1f 8b 08 00 表明收到的是 gzip 压缩文件，而不是预期的 WASM 文件魔术字节 00 61 73 6d
Searched 3 times, thought for 20s, and read vite.config.ts and utils.ts
根据您的错误信息和代码分析，这是一个 WASM 文件 MIME 类型和压缩处理的问题。以下是解决方案：
### 问题根源
1. 错误的 MIME 类型设置：nginx 配置中 /dicomweb/ location 强制设置了 Content-Type: application/wasm，这会影响所有通过该代理的文件
2. WASM 文件被 gzip 压缩：错误信息显示收到的是 gzip 文件（魔术字节 1f 8b 08 00），而不是 WASM 文件（00 61 73 6d）

### 解决方案
需要修改 nginx/default.conf 配置文件：
```
    # 处理 WASM 文件 - 特殊配置防止被压缩和确保正确的 MIME 类型
    location ~* \.wasm$ {
        add_header Content-Type "application/wasm";
        add_header Cache-Control "public, max-age=31536000, immutable";
        # 禁用 gzip 压缩 WASM 文件
        gzip off;
        # 确保不使用预压缩的 .gz 文件
        gzip_static off;
    }
```

1. HTTP status code is not ok - WASM 文件请求失败

2. 魔术字节 3c 21 64 6f 对应 ASCII <!do，这意味着返回的是 HTML 内容（通常是 404 错误页面）

这说明 WASM 文件根本没有被找到。让我们诊断问题：

问题现在更清楚了。基于错误信息，WASM 文件请求返回了 HTML 内容（魔术字节 3c 21 64 6f 是 <!do），这意味着 nginx 的 try_files 配置将 WASM 文件请求重定向到了 index.html。
